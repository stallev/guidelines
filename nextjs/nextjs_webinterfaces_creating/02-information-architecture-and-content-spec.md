# Этап 2. Information architecture и content specification

## Цель этапа

Преобразовать результаты исследования в формальную спецификацию интерфейсов: структура страниц, пользовательские потоки, контентные блоки и конверсионная логика.

## Входные данные

- Артефакты этапа 1.
- Бизнес-ограничения, KPI и юридические требования.
- Продуктовые ограничения и roadmap релизов.

## Выходные артефакты

- Sitemap (или route map App Router).
- User flows и decision points.
- Content model по страницам.
- Page briefs для дизайна и разработки.

## Структура спецификации

### 1) Sitemap / Route map

Для каждой страницы:

- URL/slug;
- роль в воронке;
- primary/secondary intent;
- связи с соседними маршрутами;
- ожидаемый тип рендеринга (статичный/динамичный/гибрид).

### 2) User flows

Для ключевых сценариев:

- входной контекст;
- шаги пользователя;
- точки выбора;
- целевое действие;
- fallback-путь.

### 3) Content model

- ключевой тезис страницы;
- обязательные блоки и их цель;
- доказательства (цифры, кейсы, факты);
- trust-секция;
- CTA-иерархия и microcopy.

### 4) Page brief

Каждый brief включает:

- цель страницы и KPI;
- целевую аудиторию и JTBD;
- порядок блоков;
- требования к контенту и длине;
- SEO-поля (`title`, `description`, `H1-H3`);
- внутреннюю перелинковку;
- требования к structured data (если применимо).

## People-first и conversion требования

- Каждый блок отвечает конкретному пользовательскому вопросу.
- Основной CTA видим без лишнего прокрутки в рамках выбранной композиции.
- Формы минимизируют friction (минимум обязательных полей, понятные ошибки, прозрачные ожидания).
- Контент избегает пустых обещаний и подтверждается фактами.

## Критерии качества этапа

- Для каждой страницы подготовлен валидный page brief.
- Route map согласована с user flows и KPI.
- Контентная модель покрывает обязательные trust и CTA-блоки.
- Спецификация готова для передачи в `.pen` без неоднозначностей.

## Definition of Done

- Готовый пакет IA+content можно напрямую использовать для создания `.pen` макетов.
