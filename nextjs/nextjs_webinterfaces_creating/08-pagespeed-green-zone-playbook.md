# Этап 8. Google PageSpeed green-zone playbook

## Цель

Сделать выход в green zone обязательным результатом pipeline для ключевых страниц.

## KPI и пороги

Обязательные целевые значения:

- PageSpeed (mobile): `>= 90`
- PageSpeed (desktop): `>= 90`
- Core Web Vitals (p75):
  - `LCP <= 2.5s`
  - `INP <= 200ms`
  - `CLS <= 0.1`

Дополнительно рекомендуется контролировать:

- TTFB (в разумных пределах для выбранного runtime/infra)
- размер JS на route
- критический rendering path

## Фреймворк измерений

Используйте минимум два уровня контроля:

1. Lab: Lighthouse/PageSpeed на representative URL.
2. Field: CrUX и/или RUM через `web-vitals`.

Правило: релиз блокируется, если lab и field сигналы указывают на устойчивую деградацию относительно порогов.

## Оптимизация по метрикам

### LCP

- Оптимизировать hero-секцию и LCP-кандидат (изображение/текстовый блок).
- Использовать `next/image` с корректными размерами и приоритетом для критичного контента.
- Не перегружать above-the-fold тяжелыми скриптами и эффектами.
- Минимизировать задержки ответа и лишние data dependencies на критическом пути.

### INP

- Снижать объем и сложность client-side JavaScript.
- Разбивать тяжелые задачи и избегать long tasks.
- Минимизировать стоимость обработчиков событий в высокочастотных интеракциях.
- Держать Server Components как default, выносить в client только действительно интерактивные части.

### CLS

- Всегда резервировать место под медиа, dynamic blocks и embeds.
- Стабилизировать размеры контейнеров skeleton/loading состояний.
- Избегать поздних вставок контента над уже отрисованным регионом.

## Next.js-specific requirements

- Использовать `next/font` и не создавать блокирующую загрузку шрифтов.
- Применять корректную cache strategy (`force-cache`, `no-store`, `revalidate`) по типу данных.
- Использовать streaming и `Suspense` для асинхронных регионов с явными fallback.
- Держать client boundaries минимальными; не делать верхнеуровневые layout полностью client-side.
- Включать bundle/chunk анализ для страниц с риском деградации.

## Связка motion и performance

- Анимации не должны конфликтовать с LCP/INP/CLS целями.
- Для насыщенных интерфейсов обязательно `prefers-reduced-motion`.
- Не анимировать крупные layout-перестройки на начальной загрузке страницы.

## Workflow: diagnose -> fix -> verify

1. Снять baseline метрики (mobile/desktop).
2. Приоритизировать проблемы по влиянию на LCP/INP/CLS.
3. Внести исправления малыми итерациями.
4. Повторить измерение тем же профилем.
5. Зафиксировать итоговый отчет и решение `pass/requires-fix`.

## Антипаттерны

- Оптимизация только desktop-версии без контроля mobile.
- Чрезмерный client-side рендеринг контента, который может быть серверным.
- Декоративные анимации в критическом above-the-fold контенте.
- Бесконтрольные сторонние скрипты и виджеты.

## Release gate

Страница может перейти к релизу только при одновременном выполнении:

- PageSpeed `>= 90` (mobile/desktop) для целевых страниц.
- Core Web Vitals пороги соблюдены.
- Нет блокирующих regressions по a11y и UX-состояниям.
